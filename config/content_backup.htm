課程簡介:

cad2025 課程將以每組至多六人分組進行，從繪圖範圍 20cm x 20cm 的 5-bar Linkage Plotter 與 Openduck mini 零組件設計繪圖作為開端，各組成員可以自選 CAD 套件完成。其中 Plotter 驅動設計建議採用 NEMA 17 步進馬達 (NEMA 17 Onshape) 或自選伺服馬達進行設計繪圖，而 Openduck mini 驅動則建議採用 STS3215 磁編碼數位伺服機 (STS3215 Onshape) 進行設計繪圖。

5-bar Linkage Plotter 與 Openduck mini 零組件設計繪圖完成後，將設法利用 Webots 進行控制系統設計與動態模擬 (Dynamic Simulation)，課程最後將針對此兩項機電系統的實作進行規劃與討論。

portable_2026.7z (387MB)

解開壓縮後，雙點擊 start_miniconda3.bat 啟動可攜系統，雙點擊 stop.bat 關閉可攜系統。

可攜系統啟動後，在命令列執行 conda activate y:\envs\cmsimde 可進入能執行 CMSiMDE 的環境。

若希望可攜系統啟動直接進入 cmsimde 環境，可將 start_miniconda3.bat 檔案中第 98 行: call "%CONDA_ROOT%\Scripts\activate.bat" %ENV_NAME% 更換為 call "%CONDA_ROOT%\Scripts\activate.bat" %Disk%:\envs\cmsimde。

cad_data.7z (包括 NX2312、Blender4.2 與 Webots2023b，4.2GB)

解開壓縮檔，將 nx、Webots_2023b 與 Blender 4.2 等目錄放入可攜系統的 data 目錄中。

在命令列執行 webotsw 可啟動 Webots，設定 NX 使用授權後，執行 ugraf -nx 可啟動 NX2312。

Webots2023b 啟動時需要 openssl_lh_set_thunks 函式，因為這是特別的 OpenSSL 介接，其目的可能是透過此函式與 OpenSSL 整合記憶體分配、除錯、或 OpenSSL 引擎擴充。

目前已知 miniconda 中必須透過 conda install openssl=3.0.16 安裝舊版後的 openssl-3-x64.dll 中才會包含openssl_lh_set_thunks 函式。

Solvespace：

實際案例：

基本描述：Solvespace其優勢為免安裝&非常輕量，但缺點是沒有誘人的介面,繪製時比較難正視座標(不排除未找到的可能性)，對初學者不友好。

Siemens NX：

實際案例：

基本描述：常用於航太、汽車、船舶等領域，但其功能很多，不管是加工、製圖或是分析都可以使用，但是對於毫無經驗的使用者不友好。

Onshape：

實際案例：plotter初步設計

基本描述：起初從影片範例去分析他的構造再來用onshape去繪製各個部件考量到有最長伸長距離問題，所以每個部件之間的間距需要去設計。

SolidWorks：

實際案例：根據設計本次的連桿機構來說，使用Solidworks的最大優勢除了程式應用廣泛外，後續如果需要進行有限元分析，操作會比較簡單。

基本描述：在業界&學界使用十分廣泛，因其軟體為買斷式且提供Simulation&Flow Simulation等，同一軟體整合多種分析功能。

Inventor：

實際案例：本次設計連桿在Inventor的優勢僅為教育使用權索取較為容易&操作容易之外並沒有甚麼特別。但因原廠Autodesk公司開發許多軟體，在配合同家產品使用時當然問題會比較少。

基本描述：Inventor使用者也不算少數，但因其採租約制，所以沒有Solidworks這麼有名，其教育授權是我目前認為速度快且方便的軟體商之一，所以從Inventor入門也是選擇之一。

Q1.Webots 是什麼？

A1.webots是一個常用於機器人與汽車自動駕駛的模擬軟體，他主要是使用網格體去模擬外型、碰撞等，也有camera可以做動作檢查或是影像識別，也有motor、senser，可以盡量真實還原模擬。

Q2.有什麼特點？

A2.webots相較於nvidia的isaac sim僅需CPU即可工作，通用性高且成本低，適合初學與練習。

特性 Webots (ODE) NVIDIA Isaac Sim (PhysX)
主要數學模型 常微分方程 (ODE) 數值求解 剛體動力學方程 LCP 求解器
核心求解器 迭代或矩陣分解法 (如 PGS) 高度優化的 Sequential Impulse/Iterative Projection
關鍵優勢 成熟、開源、通用性高 GPU 加速、高保真度、處理大量物體穩定
計算方式 主要依賴 CPU 主要依賴 GPU (CUDA) 進行並行計算

組裝的邏輯理念

1.建立Robot的原因是需要controller使用code控制，但凡遇到任何需要controller的都是建立Robot

2.children裡面的東西會跟著一起動。以您的手臂舉例，上臂與前臂是由同一關節組成，這意味著兩者相互影響，如果您要舉起手，需要將上臂往上舉，則前臂會跟隨一起往上舉，所以前臂應建立在上臂的children內。以電腦來說，因為上臂的終點是肘手關節，前臂的起點是手肘關節，因為在同一層級下則需重新計算對應位置，消耗算力十分沒有必要的，因此前臂直接建立在上臂的children內，相對於上臂控制即可。

41323107：王瑞億

openduck：

當老師提出了openduck這個專案，當下認為這個東西nvidia已經做出來了，我們為什麼還要再做一次，而且還只是用電腦去模擬。

接下來慢慢深入了解，首先在webots模擬軟體就遇到了問題，最一開始的完全不知從何開始，跟著老師的教學與網路教學，花了8個小時完全理解不同層級代表時什麼，父系子系的對應關西，也成功用作火箭的推力模擬。

製作plotter心得:

這次在 Webots 製作 Plotter 的操作流程，主要是在 Scene Tree 逐層把機構「接起來」並讓控制器能「叫得到」裝置。先把場景底板（RectangleArena、背景光源）建好，接著在 Robot 底下依序建立各段 HingeJoint，重點是把 jointParameters 的 axis、anchor、以及必要的 minStop / maxStop 檢查清楚，避免關節方向或支點位置錯誤造成整體幾何跑掉。每個關節的 device 也要確實放入 RotationalMotor，並且把 motor 的 name（如 t1、t2）固定一致，不然控制器用 getDevice() 會找不到。連桿端用 endPoint 的 Solid / SolidReference（solidName） 方式把 link 串接，配合 DEF 命名讓結構更好管理、也比較不容易接錯。最後再確認各 Solid 的 translation/rotation、rotationStep，以及 physics（密度、質量設定）與 boundingObject，確保模擬運動比較穩定。

整體心得是：Webots 這類機構作品成功與否，常常不在程式本身，而在 Scene Tree 的命名一致性、關節參數正確性、以及 link 串接關係 是否完全對得上。

plotter製作心得:

起初完全不懂webot是甚麼，這個軟體怎麼用的情況下，再組裝時可以說是有非常大的問題，後面透過老師的影片，以及上課內容以及自己上網查找，去發掘webot如何應用，再組裝時很麻煩的因該就是節點座標的輸入，每一層 jointParameters 中的 axis 與 anchor 必須與該連桿 Solid 的 translation 座標完全重合，否則模擬啟動時會因支點偏移產生劇烈抖動，在 device 位置裡面加入 RotationalMotor 並命名t1, t2，才能讓controller去透過 getDevice() 成功調用馬達，再來利用 DEF 為副鏈末端的solid命名，接著在最末端加入 SolidReference 節點並填入該 link3，讓兩段獨立的運動鏈形成穩定的閉迴路。每個 Solid 必須配置physics與 boundingObject。若連桿間的碰撞體（Bounding Object）重疊，會導致內力衝突，需微調使其略小於視覺模型。
